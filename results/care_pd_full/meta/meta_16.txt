# Analysis of Program Evaluation Results

## Successful Algorithmic Patterns

- **The current best program (Generation 13, macro_F1=0.41)** uses frozen MotionCLIP encoder with multi-window 50% overlapping 60-frame windows, attention pooling, and critically **uses `nn.CrossEntropyLoss(weight=w)` with properly computed inverse-frequency class weights** — this is the key fix over Generation 10 (0.36) which computed weights but never passed them to the loss function. This single fix improved the combined score from 0.36 to 0.41.

- **Weighted CrossEntropyLoss with inverse-frequency class weights** (Generation 13) outperforms FocalLoss with fixed alpha=0.25/gamma=2.0 (Generations 15, 20 both at 0.34). The confusion matrix for Generation 13 shows 33 correct severe predictions vs. near-zero in earlier programs, confirming that properly applied class weights provide better minority class recovery than FocalLoss alone.

- **Multi-window sliding extraction with 50% overlap (stride=30)** combined with attention pooling remains the core architectural pattern across all working programs (Generations 8, 10, 13). Generation 13 achieves balanced per-class F1 (normal=0.50, mild=0.50, severe=0.23) compared to the collapsed predictions seen in programs without proper weighting (Generations 14, 18 with 525-531/781 predictions being class 2).

- **Path A (frozen MotionCLIP) with robust Path B fallback** continues to be the only architecture producing valid predictions across all folds. Generation 13 shows consistent non-zero predictions across all 6 folds (range: 0.196–0.597), confirming the encoder is loading successfully in this configuration.

## Ineffective Approaches

- **FocalLoss with oversampling** (Generations 15, 20, both macro_F1=0.34) consistently underperforms weighted CrossEntropyLoss (Generation 13, 0.41). Despite FocalLoss being theoretically suited for imbalance, the fixed alpha=0.25 and gamma=2.0 parameters combined with class-2 oversampling created a class-2 prediction bias (439/781 and 480/781 predictions respectively), worse than the balanced predictions in Generation 13 (128/781 class-2 predictions vs. 164 actual).

- **Attention pooling with randomly initialized, untrained weights** provides no learned aggregation benefit. Across Generations 8, 10, 13, and 18, the attention pooling layer is initialized randomly and set to `eval()` mode with `@torch.no_grad()`, meaning it applies random (not learned) attention weights. Despite this flaw, multi-window diversity still helps, but the pooling itself contributes noise rather than signal.

- **Label smoothing combined with oversampling** (Generations 12, 14, 18) consistently produced worse results (0.28, 0.26, 0.24) than the current best without these techniques. Generation 18 showed the most extreme collapse (525/781 class-2 predictions), suggesting that combining oversampling with label smoothing amplifies class-2 bias rather than correcting it.

- **Programs relying on MotionCLIP checkpoint availability without robust fallback** (Generations 19: score=0.0) fail entirely. Generation 19's zero score despite being architecturally similar to Generation 13 confirms that checkpoint loading failures cascade to complete program failure when the fallback path is broken.

## Implementation Insights

- **The critical fix in Generation 13** was passing computed class weights to the loss function: `loss_fn = nn.CrossEntropyLoss(weight=w)` where `w = counts.sum() / (n_cls * counts)`. Previous programs (Generation 10) computed `w` but used `FocalLoss(alpha=0.25, gamma=2.0)` without incorporating `w`, rendering the weight computation dead code. This single-line fix accounts for the 0.05 improvement (0.36→0.41).

- **The confusion matrix for Generation 13** (169/341 normal correct, 149/276 mild correct, 33/164 severe correct) shows dramatically better severe class recovery than Generation 10 (1/164 severe correct). The prediction distribution (333 normal, 320 mild, 128 severe vs. true 341/276/164) is the most balanced seen across all evaluated programs, confirming that weighted CrossEntropyLoss successfully prevents majority-class collapse.

- **High fold-to-fold variance persists** in Generation 13 (range: 0.196–0.597, fold 4 best at 0.597, fold 2 worst at 0.196). This 3× variation between worst and best fold mirrors the pattern in Generation 10 (0.12–0.56), suggesting the variance is structural — likely from uneven severe PD case distribution across folds — rather than from the loss function choice.

- **The attention pooling layer is re-initialized randomly for every call to `extract_features`** in Generation 13, meaning different folds use different random attention weights. This is an implementation bug that adds fold-to-fold variance without providing any learned benefit. Despite this, the program still achieves the best score, indicating the multi-window feature diversity dominates over pooling quality.

## Performance Analysis

- **Score progression across correct programs**: Generation 0 (0.35) → Generation 8 (0.32) → Generation 10 (0.36) → **Generation 13 (0.41, current best)**. The jump from 0.36 to 0.41 is the largest single improvement observed, attributable entirely to fixing the class weight application rather than any architectural change.

- **Generation 13 vs. Generations 15/20 (FocalLoss variants)**: Both FocalLoss programs scored 0.34 vs. Generation 13's 0.41, a 21% relative improvement for weighted CrossEntropyLoss. This is particularly notable because Generations 15 and 20 added oversampling on top of FocalLoss, which should theoretically provide stronger imbalance correction but instead worsened performance.

- **The 0.41 score remains 27% below the RandomForest baseline (0.565)**, confirming that frozen MotionCLIP features with shallow MLP classifiers are fundamentally insufficient regardless of loss function tuning. The severe class F1 of 0.23 (33/164 correct) is the primary bottleneck — improving this single class from 0.23 to ~0.60 would push the combined score above the baseline.

- **Programs with broken implementations (Generations 16, 19: score=0.0)** continue to represent 2 of the 8 programs in this generation batch, maintaining the ~25% failure rate observed in previous generations. Generation 16 failed due to missing `def` header (syntax error), and Generation 19 failed due to checkpoint/fallback path issues, consistent with the pattern that implementation correctness rather than algorithmic sophistication is the primary failure mode.

# META RECOMMENDATIONS

The following are actionable recommendations for the next program generations:

1. **Fix the attention pooling bug by making it a persistent, trained module rather than re-initializing it randomly per call.** Move `attention_pool` outside of `extract_features` and instead store raw per-window embeddings as features (shape `(N_windows, 512)`), then define a combined `AttentionPoolMLP` model inside `train_classifier` that learns attention weights jointly with the MLP classifier. This converts the current random-noise aggregation into a genuine learned temporal aggregation — the multi-window diversity already helps (Generation 13 beats single-window), so adding learned pooling on top should provide additional discriminative signal, particularly for the severe class where gait freezing episodes may appear in only some windows.

2. **Add a RandomForest classifier trained on the same 512-dim MotionCLIP features and ensemble its `predict_proba` output with the MLP's softmax probabilities.** The RandomForest baseline achieves 0.565 macro-F1 on raw features vs. the current MLP's 0.41 on MotionCLIP features, suggesting tree-based methods capture complementary discriminative patterns. Import `sklearn.ensemble.RandomForestClassifier` with `n_estimators=200, class_weight='balanced'`, train it on `feat_tr`, then average `rf.predict_proba(feat_te)` with `softmax(mlp_logits)` using equal weights before taking argmax — this is a low-risk addition that preserves the working MLP pipeline while adding the tree-based signal that dominates the baseline.

3. **Concatenate velocity features (frame-to-frame differences of 6D rotations) to the MotionCLIP embedding before the MLP classifier.** Compute `vel = np.diff(rot6d, axis=0)` for each window, take `mean` and `std` across frames to get a 300-dim velocity summary, then concatenate this to the 512-dim MotionCLIP embedding to form a 812-dim input to the MLP. PD gait is clinically characterized by reduced movement velocity and freezing episodes — these dynamics are discarded by the current mean-pooled MotionCLIP embedding but are directly captured by first-order differences, and this addition requires only minor changes to the existing working pipeline.

4. **Unfreeze only the last 2 transformer encoder layers of MotionCLIP and fine-tune them jointly with the MLP at a very low learning rate (1e-5) while keeping earlier layers frozen.** The current frozen MotionCLIP features are demonstrably insufficient (0.41 vs. 0.565 RandomForest baseline), and the pretrained weights were trained on diverse motions rather than PD gait specifically. Use two parameter groups: `{'params': enc.encoder.layers[-2:].parameters(), 'lr': 1e-5}` and `{'params': clf.parameters(), 'lr': 1e-3}`, training for 100 epochs with the same weighted CrossEntropyLoss — this domain adaptation is the most direct path to surpassing the baseline since it allows the encoder to adapt its representations to PD-specific gait patterns.

5. **Replace the single MLP with a deeper residual MLP (3 residual blocks) and increase training epochs to 300 with a warmup schedule.** The current 2-hidden-layer MLP (512→256→128→3) may be underfitting the 512-dim feature space given only 150 epochs — add residual connections `x = x + layer(x)` with layer norm between blocks, and implement a linear warmup for the first 30 epochs followed by cosine annealing. Keep the proven `nn.CrossEntropyLoss(weight=w)` with inverse-frequency weights unchanged, as this is the single most impactful component in the current best program, and focus the architectural change purely on classifier capacity and training stability.
